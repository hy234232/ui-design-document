# 핸드오프문서 자동생성기 — 완전 재현 법칙 (Spec-of-Record)

> **플러그인 정식 명칭:** 핸드오프문서 자동생성기 (manifest.json `name`).
> **목적:** 이 문서 하나만 그대로 따르면 **어떤 Figma 플러그인에서도 동일한 핸드오프 문서 결과물**이 나오도록 전체 파이프라인을 규정한다.
>
> **디자인시스템 단일 출처(Single Source of Truth):** [디자인시스템.md](디자인시스템.md) 파일이 모든 컴포넌트·상태·문구·단위·색상의 표준 정의를 갖는다. server.js는 매 분석 요청마다 이 파일을 다시 읽어 프롬프트에 `## 디자인시스템 정의` 섹션으로 주입하므로, 파일을 수정하면 다음 분석부터 즉시 반영된다. 토스트·모달·인풋·차트·인사이트·단위 등의 표준 문구는 이 파일에서 관리하고, server.js의 buildPrompt 내부 하드코딩된 규칙(토스트 3초, 모달 닫기 3종 등)은 백업 역할.
>
> **탭 구조 (4개, 순서는 디자인 워크플로우 순):**
> 1. 🗺️ **IA로 화면 그리기** — IA(사이트맵) 이미지 업로드/페이스트 → Claude가 leaf 페이지 타이틀 모두 추출 → Mobile/PC 라디오 + 사이즈 드롭다운(M:340/360/430, P:1024/1280/1440/1920) 기준으로 빈 프레임 N개 + 위쪽 타이틀 텍스트를 4열 그리드로 자동 생성. server 엔드포인트: `POST /extract-ia`.
> 2. 🔀 **화면 흐름 생성** — 두 프레임 사이에 ELBOWED 라운드 화살표(시작 ●, 끝 →) 자동/수동 생성. 컬러 RGB 피커 + #hex 인풋 + 자동 생성 ON/OFF 체크박스.
> 3. 📋 **기능 명세 생성** — `mode=spec`: overview + features만 도출. 옵션: 화면 설명·공통기능·이전 분석 누적.
> 4. ⚠️ **예외 케이스 생성** — `mode=exception`: Edge Case / Error Case / Corner Case 각 체크박스로 선택 생성. Edge=경계값, Error=실패·오류(구 언해피플로우), Corner=희귀 조합·복합 조건.
> 입력(선택 프레임)이 같으면 출력(화면 설명 + 기능별 FE/BE 명세 + 캔버스 그림)이 동일해야 한다.
> 이 문서는 "권고"가 아니라 **법칙(law)** 이다. 각 단계의 임계값·정규식·프롬프트 문구를 임의로 바꾸면 결과가 달라진다.
>
> **유지보수 규칙(필수):** 코드(`code.js`/`server.js`/`ui.html`)나 프롬프트 규칙이 바뀌면 **이 문서와 `figma-page-reader.md`를 같은 작업에서 즉시 최신화**한다. 법칙이 바뀌었는데 문서가 안 바뀌면 재현성이 깨진다. (이 두 md는 항상 코드와 동기 상태를 유지한다.)
>
> **최근 갱신 반영:** 비전(이미지) 병행 독해 · 화면 설명(overview)+체크박스 · 모달/딤 감지 · 모달 닫기 5종(취소·X·딤클릭+딤해제+롤백) · 토스트 fade 3초 · 테이블+스크롤 무한스크롤 기본 · 완전성(누락 금지)+자가 점검 · 비활성 필드 "읽기 전용 표시"로 언급 · 생성 메타는 md에만 기록(캔버스 미표기) · cwd/보안 고정 · 타임아웃 240초 · **핑크 번호 뱃지(anchor) — 선택 프레임 복제 후 기능 위치를 화면 위에 1·2·3 핑크 뱃지로 표시, 기능정의서 번호와 1:1 매칭** · **언해피플로우(예외·실패)·엣지케이스 섹션 — 체크박스로 토글, features와 동일 카드 형식(아이콘+번호+FE/BE), 섹션별 컬러(빨강·노랑)** · **PRD/요구사항 첨부 시 30000자까지 전체 반영(8000→30000)** · **이전 분석 누적(중복 방지) — 파일 단위 clientStorage에 기능명 누적, 체크박스 ON 시 Claude에게 "이미 정의된 기능 목록" 전달하여 GNB·사이드바·반복 요소 중복 생성 금지, 초기화 버튼 제공**.

## 0. 전체 아키텍처 (4단계 파이프라인)

```
[Figma 선택 프레임]
   │  ① 독해(code.js, 샌드박스)  — 레이어 구조 + PNG 이미지 추출
   ▼
[frameData(JSON) + 화면 PNG(base64)]
   │  ② 브릿지 전송(ui.html iframe → localhost:3765)
   ▼
[Node 브릿지 서버(server.js)]
   │  ③ 프롬프트 조립 + 이미지 저장 + Claude Code CLI 호출(비전)
   ▼
[Claude 응답: JSON 객체 {overview, features}]
   │  ④ 안전 파싱 → 캔버스 그리기(code.js)
   ▼
[캔버스: 화면 설명 + 기능별 FE/BE 명세 영역]
```

핵심 원리 2가지(둘 다 반드시 지켜야 동일 결과):
1. **레이어 구조 독해 + 화면 이미지(비전) 병행** — 레이어 이름이 자동 생성(`Rectangle 6643`)이라 의미가 없을 때, 실제 렌더 이미지를 Claude가 직접 보고 글자·버튼·표·모달을 읽는다.
2. **앵커링(anchoring)** — 화면에 실재하는 UI 요소에서만 기능을 도출한다. 도메인 상식으로 기능을 지어내지 않는다.

## 1. 1단계 — 프레임 독해 (code.js, Figma 샌드박스)

### 1.1 샌드박스 JS 제약 (반드시 준수)
Figma `code.js`는 제한된 JS 환경이다. 위반 시 플러그인이 죽는다.
- 옵셔널 체이닝 `?.` **금지** → `x && x.y`
- 옵셔널 catch 바인딩 `catch {}` **금지** → `catch (e) {}`
- `var` / 함수 선언 자유롭게 사용
- `ui.html`은 iframe(완전한 브라우저)이라 `?.` 허용

### 1.2 출력 데이터 계약 (frameData 스키마)
각 선택 노드마다 `buildFrameSummary(node)`가 아래 객체를 반환한다. **이 키 구조가 곧 계약이다.**

```jsonc
{
  "frameName": "string",
  "activeNav": "string|null",        // 사이드바/GNB 활성 메뉴명 (기능 아님, 맥락 힌트)
  "titles": [], "tabs": [], "sections": [],          // Tier 1 목적
  "buttons": [], "iconActions": [], "links": [],     // Tier 2 액션
  "inputs": [], "selects": [], "disabledFields": [], // Tier 3 데이터
  "tableHeaders": [], "listItems": [], "uploads": [],
  "hasTable": false, "hasScroll": false,             // 테이블·스크롤(무한스크롤 기본)
  "checkboxes": [], "radios": [], "toggles": [],     // Tier 4 상태
  "badges": [], "emptyStates": [], "alerts": [], "toasts": [], // 토스트=고정 규칙(3초+fade)
  "modals": [], "hasDimOverlay": false, "hasModal": false, // Tier 5 플로우(모달)
  "drawers": [], "steppers": [], "paginations": [],
  "componentVariants": [],                            // Tier 6 메타
  "allTexts": [],                                     // 도메인 언어
  "image": "base64 PNG"                               // 비전 분석용 (전송 후 서버에서 파일로 저장)
}
```

### 1.3 신호 계층 (Tier) — 레이어 이름 정규식 매칭
`collectContext(node, result, depth)`가 트리를 순회(`depth > 30`에서 중단)하며 `node.name`(소문자)으로 분류한다.

| Tier | 대상 | 정규식 (소문자 매칭) |
|---|---|---|
| 0 | **제외** 사이드바/GNB | `sidebar|side.?bar|사이드바|gnb|global.?nav|lnb|local.?nav|header.?nav|top.?bar|topbar` |
| 1 | 타이틀 | `title|heading|h1|페이지명|타이틀` |

| 1 | 섹션 | `breadcrumb|section|섹션|경로` |
| 2 | 버튼 | `button|btn|버튼|\bcta\b|\bfab\b` |
| 2 | 아이콘버튼 | `icon.?btn|icon.?action|edit.?icon|delete.?icon|copy.?icon|more.?icon` |
| 2 | 링크 | `\blink\b|anchor|더보기|자세히보기` |
| 3 | 인풋 | `input|field|textfield|인풋|텍스트필드|search|검색` |
| 3 | 셀렉트 | `select|dropdown|combobox|셀렉트|드롭다운` |
| 3 | 테이블컬럼 | `table.?header|col.?header|\bth\b|thead` |
| 3 | 리스트아이템 | `list.?item|listitem|\brow\b|card.?item|\bitem\b` |
| 3 | 업로드 | `upload|attach|업로드|첨부` |

> **법칙 0-1:** 사이드바/GNB(`isNavShell`)는 매칭되면 **즉시 하위 탐색 중단**하고 활성 메뉴명만 `activeNav`에 기록한다. 사이드바 내부 버튼·메뉴는 절대 기능으로 만들지 않는다.

> **주의(버그 회귀 금지):** 버튼 정규식의 `cta`는 반드시 `\bcta\b`(단어 경계)로 둔다. 경계가 없으면 "re**cta**ngle"에 오매칭되어 모든 Rectangle이 버튼이 된다.

### 1.4 내용·비주얼 기반 보완 독해 (`classifyByContent`) — 가장 중요
레이어 이름이 자동 생성이라 Tier 매칭이 안 될 때, **형태(fill/stroke/radius/height) + 내부 텍스트**로 역할을 추론한다.

상수:
```
ACTION_RE = /초대|추가|삭제|편집|수정|등록|저장|취소|확인|전송|발송|보내기|내보내기|가져오기|업로드|다운로드|복사|이동|변경|적용|생성|만들기|로그인|로그아웃|가입|신청|제출|승인|거절|반려|해제|연결|연동|초기화|공유|복제|미리보기|첨부|시작|중지|재시도|새로고침|결제|구매|예약|취소하기|닫기|뒤로|다음|이전|완료|보기/
PLACEHOLDER_RE = /입력|선택하세요|선택해주세요|을 입력|를 입력|검색어|예\s*:|placeholder|hint|@/
CHEVRON_RE = /chevron|caret|arrow.?down|expand.?more|드롭|down.?icon|▾|⌄|expand_more/
```

감지 규칙 (대상: FRAME/INSTANCE/COMPONENT/GROUP, 배경 또는 보더 필수, 내부 텍스트 필수):
| 대상 | 형태 | 텍스트 | 결과 |
|---|---|---|---|
| 셀렉트 | 높이 24~72px + chevron 자식 | 단일 값 ≤24자 | `selectFields` |
| 버튼 | 높이 24~72px | 짧은 텍스트 ≤14자(노드 ≤2개) + `ACTION_RE` + chevron 없음 | `actionButtons` |
| 인풋 | 높이 24~72px | `PLACEHOLDER_RE` 일치 | `inputFields`(비활성이면 `disabledFields`) |

병합: `buttons = 이름기반 ∪ actionButtons`, `inputs = 이름기반 ∪ inputFields`, `selects = 이름기반 ∪ selectFields`.

### 1.5 비활성(disabled) 감지 (`detectDisabled`) — 입력 기능 오탐 방지
다음 중 하나라도 해당하면 읽기 전용으로 보고 `disabledFields`로 분리(입력/상태전환 기능 금지):
- 이름에 `disabled|disable|readonly|read.?only|비활성|inactive|grayed|dimmed|locked|잠금`
- `node.opacity < 0.55`
- 컴포넌트 variant 값에 `disabled|readonly|inactive|비활성`
- 배경이 저채도 회색(채도 < 0.12, `isGrayish`)이고 보더 없음

> **법칙 1.5-1 (비활성 필드 언급 의무):** 비활성 필드는 **수정 기능을 만들지 않되, 존재 자체는 반드시 명세에 언급**한다. "입력"이 아니라 **"읽기 전용 조회/표시"** 기능으로 적고, 해당 필드가 **비활성(읽기 전용)으로 표시된다**는 점을 분명히 한다.
> - 예: "기본 정보"의 회사명이 비활성(회색) → **회사 기본정보 조회(읽기 전용)** / FE: 회사명 등을 비활성(읽기 전용)으로 표시, 수정 불가 / BE: 회사 기본정보 조회 API
> - (과거 회귀 주의) 비활성 필드로 "○○ 정보 입력 / 입력 전·후 상태 전환" 같은 입력 기능을 만들면 안 되지만, 아예 빼먹어서도 안 된다.

### 1.6 모달·딤 처리 감지 (`detectModalParts`) — 레이어 이름만으로는 못 잡는다
**① 딤(Dim) 오버레이** — 다음을 모두 만족하거나 이름에 `dim|overlay|backdrop|scrim|딤|배경막|오버레이`:
- 화면(루트) 대비 폭·높이 ≥ 80%
- 어두운 fill(밝기 = 0.299R+0.587G+0.114B < 0.35)
- 반투명(`node.opacity < 0.92` 또는 `fill.opacity < 0.92`)
→ `hasDimOverlay = true`

**② 모달 카드** — 이름에 `modal|dialog|popup|sheet|모달|팝업|다이얼로그|바텀시트`가 있거나, 또는:
- 밝은 컨테이너(밝기 > 0.82)
- 라운드(cornerRadius ≥ 8, 혼합 라운드 인정)
- 화면보다 작은 중앙 카드(폭 < 루트 92%, 폭 ≥ 220px, 높이 ≥ 120px)
- 내부에 제목 + 액션 버튼(`ACTION_RE`) 존재(오탐 방지: 액션 텍스트 필수)
→ `modals`에 제목 추가, `hasModal = true`

### 1.7 텍스트 독해 우선순위
- TEXT 노드의 `characters`를 모두 모아 `allTexts`로 (중복 제거, 0<길이<200).
- 의미 우선순위: ① 최상단 대형=타이틀 ② 섹션 중간=기능 경계 ③ 인풋 라벨=필드명 ④ 버튼 내부=액션명 ⑤ 뱃지=상태값.

### 1.8 화면 이미지 추출 (비전 분석 — 필수)
`get-selection` 처리 시 선택 노드마다:
```js
const bytes = await node.exportAsync({ format: 'PNG', constraint: { type: 'SCALE', value: 2 } });
frame.image = (typeof figma.base64Encode === 'function')
  ? figma.base64Encode(bytes)
  : bytesToBase64(bytes); // 폴백 수동 인코더
```
> **법칙 1-1:** 이미지 추출은 옵션이 아니다. 레이어 텍스트가 비어도 이미지로 글자를 읽어야 동일 결과가 보장된다.

## 2. 2단계 — 브릿지 전송 (ui.html)

- iframe에서 `POST http://localhost:3765/analyze`
- body: `{ frameData, prdContent|null, reqContent|null, previousFeatures: string[], includeCommon: bool, mode: 'spec'|'exception', requestEdge?: bool, requestError?: bool, requestCorner?: bool }` (frameData에 `image`·`nodeId` 포함)
- 응답: `{ features, overview, edgeCases, errorCases, cornerCases, meta }` — mode에 따라 일부 키만 채워지고 나머지는 빈 배열.
- 응답: `{ features, overview, unhappyFlows, edgeCases, meta }`
- **체크박스 3종** (모두 분석 시점에 항상 생성됨 — 체크는 표시 여부만 결정):
  - `#optOverview` (기본 체크): 화면 설명(overview) 표시
  - `#optUnhappy` (기본 미체크): 언해피플로우(예외·실패) 섹션 표시
  - `#optEdge` (기본 미체크): 엣지케이스 섹션 표시
  - `#optAccumulate` (기본 체크): **이전 분석 누적 — 새로 등장한 것만 명세**. 같은 파일에서 페이지를 바꿔가며 분석할 때 GNB·사이드바·지점셀렉트·기간셀렉터 등 공통 요소를 매번 중복 생성하지 않도록 함. 체크 ON 상태로 분석 성공 시 자동으로 새 기능명들이 누적 저장됨. 체크 OFF로 분석하면 누적되지 않고 그 분석은 전체 기능을 다 뽑음.
  - 누적 카운트(`#histCount`)는 분석 직후 갱신, 초기화 버튼(`🗑️ 초기화`)으로 파일별로 리셋 가능.
  - `#optCommon`/`#optCommonE` (기본 미체크) **🧱 공통기능 읽기**: GNB·사이드바·설정·프로필 등 공통 영역을 명세 대상에 포함. ON 시 `code.js`의 `isNavShell` 단락이 우회되어 하위 트리가 정상 탐색됨. server 프롬프트도 분기 — OFF면 "사이드바·GNB 제외" 규칙, ON이면 "공통 기능 명세 모드"로 GNB 메뉴 이동·지점 셀렉트·설정·프로필 같은 공통 액션을 명시적으로 도출하라고 지시.
  - **화면 흐름 탭 전용**: `#optAutoFlow`(기본 ON) — 2개 프레임 선택 시 즉시 화살표 그림. `#arrowColor`(RGB 피커) + `#arrowHex`(텍스트 입력) 양방향 동기화. 변경 시 `set-flow-options` 메시지로 code.js 상태 갱신. "선택한 두 프레임 연결" 버튼은 `draw-flow-now` 메시지.
  - **예외 케이스 탭 전용**: `#optEdgeCase`(기본 체크) / `#optErrorCase`(기본 체크) / `#optCornerCase`(기본 미체크) — Claude에 요청 플래그(`requestEdge/Error/Corner`)로 전달, 체크된 케이스만 생성됨.
  - 모든 체크박스의 `onchange` → `onToggleSections()` → `renderFeatures()` 즉시 재렌더.
  - 캔버스: `draw-spec` 전송 시 체크된 것만 포함 (`overview: 체크? lastOverview : null`, `unhappyFlows: 체크? lastUnhappy : []`, `edgeCases: 체크? lastEdge : []`).
- **PRD/요구사항 첨부**: 첨부 시 그 내용을 처음부터 끝까지(최대 30000자) 프롬프트에 포함, server는 반드시 읽고 화면 요소와 1:1 대조해 반영하도록 지시받음.

### 3.3 프롬프트 구조 (순서 고정 — 동일 결과의 핵심)
1. **역할**: "경력 20년 시니어 PM"
2. **화면 이미지 섹션(최우선)**: `@경로`로 이미지 첨부 + "Read 도구로 직접 열어 버튼 글자·placeholder·표 컬럼/셀·드롭다운 선택지·타이틀·**모달(딤+카드)** 을 눈으로 읽어라. 레이어 구조는 보조자료."
3. **핵심 원칙**: 화면에 실재하는 UI에서만 도출, 지어내기 금지.
4. **사이드바·GNB 규칙**: `현재 위치:` 항목은 기능 아님.
5. **액션 우선 규칙**: 핵심 기능은 버튼·셀렉트 동작에서. 정적 라벨로 "○○ 정보 입력" 만들지 마라.
6. **비활성 필드 규칙**: `[데이터·비활성]`은 읽기 전용, 입력/상태전환 금지.
7. **모달 규칙**: 딤 배경 + 중앙 카드 = 모달. 발견 시 **반드시 "모달 구현" 기능 포함**. 모달 뒤 가려진 본문으로 중복 기능 금지.
   - **모달 닫기 동작은 항상 3종을 모두 명세에 적는다(예외 없음):** ① 취소/닫기 버튼 — **화면의 실제 라벨**(`취소`·`닫기`·`아니오` 중 보이는 것 그대로, 지어내지 말 것) ② 우상단 X 아이콘 버튼 ③ 모달 외 영역(딤 배경) 클릭.
   - 모달을 닫을 때는 **딤 배경(오버레이)도 함께 해제(제거)** 한다는 점을 항상 명세에 적는다.
   - 닫을 때 입력/변경 중 내용이 있으면 **필요 시 변경 사항 롤백** 처리도 FE 할일에 포함.
   - 모달 FE 할일 표준형: `모달 표시/딤 오버레이 + (취소/닫기/아니오) 버튼·X 버튼·딤 배경 클릭으로 닫기 + 닫을 때 딤 배경 해제 + 필요 시 변경 롤백 + 포커스 트랩·폼 검증`.
   - (예외) 모달 FE 할일은 닫기 3종을 담느라 40자 제한을 넘겨도 된다.
7-1. **토스트 알림 규칙(고정)**: 화면에 토스트/스낵바가 있거나 액션 성공/실패 피드백이 필요하면 토스트를 명세에 포함한다.
   - 토스트 동작은 항상 고정: **3초 뒤 자동으로 사라지며, 서서히 나타났다가(fade-in) 서서히 사라진다(fade-out)**.
   - 토스트 FE 할일 표준형: `토스트 알림 표시 — fade-in 후 3초 뒤 fade-out으로 자동 사라짐`.
   - 토스트는 `toasts` 신호(이름 `toast|snackbar`)로 감지되어 `[상태] 토스트/스낵바`로 프롬프트에 전달된다.
7-2. **테이블·스크롤 규칙(고정)**:
   - 테이블/목록(`hasTable`, 테이블컬럼, 리스트아이템)이 있으면 **목록 조회/표시 기능**을 명세에 포함한다.
   - 스크롤(`hasScroll`)이 있고 **"한 번에 몇 개씩 불러오는지"(페이지 크기) 설명이 화면·문서에 없으면 → 기본값 "무한 스크롤(infinite scroll)"** 로 명세한다.
     - FE: 스크롤 끝 도달 감지 → 다음 페이지 lazy load + 로딩 인디케이터.
     - BE: 커서/offset 기반 페이징 조회 API.
   - 단, 명시적 페이지네이션이나 "N개씩 보기" 표기가 있으면 그 방식(페이지 기반)을 따르고 무한 스크롤로 가정하지 않는다.
   - 감지: 이름 `table|테이블|grid|목록|리스트뷰` 또는 `tableHeaders` 존재 → `hasTable`. 프레임 `overflowDirection !== 'NONE'` 또는 이름 `scroll|스크롤` → `hasScroll`.
8. **화면 정보(pageContext)**: Tier별 `### [목적/액션/데이터/상태/플로우] …` 라인 (빈 항목 제외). `[플로우·모달] 딤 감지`, `[데이터·비활성]` 포함.
9. **작성 규칙**: 기능명=동사+목적어(화면 텍스트 그대로), FE/BE 각 40자 이내, 장식 UI 제외.
9-1. **완전성(누락 금지) 규칙 — 기능 누락의 핵심 방지책**:
   - 화면의 **모든 상호작용 요소를 빠짐없이** 기능으로 다룬다. 다음은 각각 최소 1개 기능에 대응되어야 한다: 모든 [액션] 버튼 · 모든 [데이터] 셀렉트/드롭다운(선택 동작 자체도 별도 기능 가능) · 모든 입력+버튼 조합 · 테이블/목록 · 비활성(읽기 전용) 필드 · 모달 · 토스트.
   - 여러 요소를 임의로 한 기능에 뭉뚱그려 다른 요소를 누락시키지 않는다. 동작이 다르면 분리한다.
   - **출력 직전 자가 점검(필수):** 각 요소가 features에 최소 1번 등장하는지 확인하고 빠진 건 추가한 뒤 출력한다(점검 과정은 출력하지 않음).
   - **배경:** 기능 누락의 주원인은 LLM 비결정성 + 완전성 규칙 부재였다. 이 규칙으로 실행 간 누락(예: "초대 권한 선택"이 "구성원 초대"에 합쳐져 사라짐)을 방지한다.
10. **레이블 없는 버튼 처리**: 이름이 `Rectangle 1234`여도 `[텍스트]`/이미지 도메인 단어로 추론, 포기 금지.

## 4. 4단계 — 출력 계약 & 파싱 & 캔버스

### 4.1 출력 형식 (JSON 객체 — 첫 글자 `{`, 끝 글자 `}`)
```json
{
  "overview": {
    "title": "화면 이름 (예: 회사 구성원 관리)",
    "purpose": "이 화면이 무엇을 하는 화면인지 1~2문장"
  },
  "features": [
    {
      "feature": "기능명(동사+목적어)",
      "icon": "이모지1개",
      "fe": "FE 할일 40자내",
      "be": "BE 할일 40자내",
      "anchor": { "x": 0.42, "y": 0.18, "note": "트리거 UI 짧은 설명" }
    }
  ]
}
```

**anchor 규칙 (필수):**
- 각 feature에 `anchor: { x, y, note }` 또는 `anchor: null` 포함.
- `x`, `y`는 0~1 사이 실수 — 이미지 좌→우, 상→하 비율(트리거 UI 요소의 시각적 중심점).
- `note`는 어떤 UI인지 짧은 설명(예 "기간 셀렉트", "초대 버튼").
- 인풋+버튼 조합은 보통 **버튼 위치** 사용.
- 화면 전반에 걸치거나 위치 특정 어려우면 `null` (그래도 기능은 포함).
- 이미지를 직접 보고 좌표를 측정해 0~1 비율로 정밀하게 환산 — 대충 찍지 말 것.
규칙:
- 산문·머리말·코드펜스 금지. JSON 객체 하나만.
- `[텍스트]·[액션]·[데이터]·이미지`에 초대/권한/구성원/이메일/소유자·관리자·일반회원/내보내기/해제 등 도메인 단어가 하나라도 있으면 `features` 빈 배열은 **오답**.
- 인풋(이메일) + 액션버튼(초대) 조합은 명백한 기능 → 누락 금지.

### 4.2 안전 파싱 (`extractJson`)
- 코드펜스 제거 후, 먼저 나오는 `{` 또는 `[`를 시작점으로 **균형 괄호 스캔**(문자열 내부 괄호 무시).
- 객체면 `{overview, features}`, 배열이면 구버전 `features`로 해석(하위 호환).
- 파싱 실패 시 추출 내용 일부를 담아 명확한 에러.

### 4.3 캔버스 그리기 (`createSpecOnCanvas(features, selectionBounds, overview, meta, sourceNodeIds)`)
- **0) 선택 프레임 복제 + 핑크 뱃지 오버레이 (NEW)**: `sourceNodeIds[0]`을 `node.clone()`으로 복제하고 원본 아래 80px에 배치. 각 feature의 `anchor.x·y`(0~1 비율) × 클론 가로·세로 = 절대 좌표, 거기에 핑크 뱃지를 그린다.
  - 뱃지: 32×32, fill `#ff2d8c`, radius 6, drop-shadow(검정 25%, y=2, blur 4), 내부 흰색 Bold 18px 번호(1부터 1:1 매칭, **번호는 features 배열 인덱스+1로 기능정의서 목록 번호와 정확히 일치**).
  - 뱃지의 중심이 anchor 좌표 위에 오도록 `x = cloneX + rx*cloneW - 16`, `y = cloneY + ry*cloneH - 16`.
  - 클론 + 뱃지들을 한 그룹("`{이름} (기능 위치)`")으로 묶어 이동·정리를 쉽게 함. 그룹 생성 실패 시 wrapper 위치 계산에 클론 단독 bounds 사용.
  - anchor가 null이거나 x/y 누락이면 해당 기능의 뱃지는 그리지 않음(기능 자체는 명세에 그대로 포함).
- 래퍼 프레임 "기능 설명": 폭 = max(선택폭, 1024), 배경 `#363b46`, 보더 `#727786`, radius 20.
  - 클론이 있을 때: 클론 그룹 아래 24px.
  - 클론이 없을 때: 기존대로 선택 프레임 아래 80px.
- **언해피플로우·엣지케이스 섹션 (NEW)**: features 카드 모두 그린 다음, 섹션이 비어있지 않으면 굵은 구분선(`#727786`, 2px) → 좌측 컬러 바(4×32) + 26px Black 섹션 타이틀 → features와 동일한 카드 반복(번호 뱃지 색상만 변경) → 다시 굵은 구분선.
  - 언해피: 컬러 `#ff6b6b`, 번호뱃지 배경 `#3a1d1d` · 번호 텍스트 `#ff6b6b`. 섹션 타이틀 "⚠️ 언해피플로우 (예외·실패)".
  - 엣지케이스: 컬러 `#ffc94a`, 번호뱃지 배경 `#332811` · 번호 텍스트 `#ffc94a`. 섹션 타이틀 "🧩 엣지케이스".
  - 항목명에 anchor.note가 있으면 항목명 옆에 `  · {note}` 형태로 부착해 어느 UI 요소인지 표기.
  - features의 핑크 뱃지(클론 위) 색상과 섹션 번호뱃지 색상이 일치해 시각적 연결 — features=흰색(클론에는 핑크), 언해피=빨강, 엣지=노랑.
- **개요 헤더(overview 있을 때만)**: `이 화면은`(라벨) → title(30px Black) → purpose(20px, lineHeight 30, 줄바꿈) → 굵은 구분선(2px).
- **기능 항목 반복**: ① 번호 뱃지(흰 24px) ② 기능명(24px Bold) ③ FE 행(뱃지 `#0d5215`/`#146c1e`/`#53e464`) ④ BE 행(뱃지 `#16468a`/`#2f66b6`/`#5cb8ff`) ⑤ 구분선(마지막 제외, 1px `#686e7a`).
- **메타(모델·환경)는 캔버스에 그리지 않는다.** 결과물에는 개요+기능 목록만 그린다. (메타는 아래 4.4처럼 md 문서에만 기록한다.)
- 패딩 48, 항목간격 48, 헤더→태스크 20, FE↔BE 12, 항목 하단 40.
- 폰트: Pretendard 우선(Black/Bold/Regular), 폴백 Inter(Black=Bold).

### 4.4 생성 환경 메타(meta) — md 문서에만 기록 (캔버스에는 그리지 않음)
- 모델을 **고정하지 않는다.** 대신 **어떤 모델·환경에서 생성했는지를 이 md 문서에 기록**한다. (캔버스 결과물에는 표기하지 않는다 — 사용자 요청.)
- 서버는 참고용으로 응답에 `meta`를 포함하지만(`{ model, tool, os, node, generatedAt }`), **플러그인은 이를 캔버스에 렌더하지 않는다.**
- `model`은 server.js 상수 `MODEL`에서 가져온다. 모델/환경이 바뀌면 결과 문구가 달라질 수 있으므로, 결과를 비교할 때 아래 "생성 환경 기록"을 근거로 삼는다.
